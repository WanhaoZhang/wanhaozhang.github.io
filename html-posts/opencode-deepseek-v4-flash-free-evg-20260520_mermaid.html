                        • EVG（Epilogue Visitor Graph）执行路径深度分析

                        工具：opencode 模型：deepseek-v4-flash-free 分析功能：EVG（Epilogue Visitor Graph） 生成日期：2026-05-20 分析范围：本地工作树，master 分支，Python 前端→图优化→C++ 代码生成→Ascend 硬件运行时（AtlasA2/Ascend950），含 GM workspace 和 UB workspace 两种 DispatchPolicy

                        EVG 是一个"用 Python 写 epilogue 逻辑，自动生成 Ascend 硬件 C++ 模板代码"的 DSL 框架。用户只需定义一个纯 Python 函数描述后处理逻辑（如 result = accum + bias; return result），EVG 系统自动完成：AST 解析 → DAG 建模 → 7 道图优化 Pass → C++ 类型/参数生成 → kernel 模板嵌入 → 编译为 AscendC 可执行程序。运行时采用 AIC（矩阵乘）+ AIV（后处理） 跨核分工流水线，利用硬件事件（V_MTE2/MTE2_V/MTE3_V/V_MTE3）实现双缓冲 LOAD-COMPUTE-STORE 三级流水。

                        整体调用链路

                        阶段一：API 入口层——GEMM 计划创建与 EVG 分支

                        执行流程分析

                        • 输入：用户传入 evg_config = {"fn_src": "def epilogue(accum, bias): ...", "example_inputs": {"accum": OpTensor(...), "bias": OpTensor(...)}}

                        • 关键分支：GemmKernelBase.__init__ 保存 evg 配置（gemm_base.py:50）。get_kernels() 检查 self.evg 非空时直接创建 BasicMatmulTlaVisitorKernel（包含 EVG 支持）。普通 kernel 可通过 to_evg() 转换——检查 _features["is_support_evg"]（basic_matmul_tla_visitor.py:15），若为 True 则构造同参数的 VisitorKernel 实例并传递 evg 配置（gemm_base.py:134）。

                        • 约束：仅 BasicMatmulTlaVisitorKernel 和 BasicMatmulKernel 支持 EVG（is_support_evg: True）。GroupedMatmul、SplitK、StreamK、Batched 等 kernel 不支持。

                        • 输出：渲染时 get_render_params() 调用 evg()（gemm_base.py:91）得到 callback_name, evg_args, evg_str，嵌入到 _KERNEL_TEMPLATE（basic_matmul_tla_visitor.py:53）的 evg_callback_name 位置。

                        • EpilogueDispatchPolicy：GM workspace 固定 EpilogueVisitor<false>（basic_matmul_tla_visitor.py:98），UB workspace 用 <true>（取决于 arch_tag 是否为 Ascend950，basic_matmul_tla_visitor.py:124）。

                        阶段二：EVG 前端——AST 解析与 DAG 建模

                        执行流程分析

                        • 输入：用户 Python 函数源码字符串（fn_src）和示例输入张量字典（example_inputs，OpTensor 类型）。

                        • 解析过程：PythonEVGParser 继承 EpilogueVisitorGraph（networkx DiGraph 封装）和 ast.NodeVisitor（evg_extension.py:238）。parse() 执行 ast.parse 后调用 visit，遍历 FunctionDef：

                          • 函数参数 → add_load_node：参数名为 "accum" 时设置 op=accload，其余为 auxload。若形状含 sympy.Symbol 类型，记录到 symbol_shape_substitution_dict 用于后续 m/n 替换（evg_extension.py:158）。

                          • 二元运算 → ast_op_bindings 映射 ast 算子到 EpilogueOp 枚举（evg_extension.py:254），创建 ComputeNode 并用 add_edge 连接父子节点，pos 参数标记输入位置（evg_extension.py:342）。

                          • 函数调用 → 支持 relu/leakyRelu/Prelu/max/min/sigmoid/silu、cast（添加 CastNode）、constant（添加 ConstantNode）。

                          • 赋值 → 左值为 StoreNode（evg_extension.py:346）。

                          • return → 标记输出 StoreNode 的 is_output=True，设置 element/shape/stride（evg_extension.py:137）。

                        • 节点创建接口：add_load_node、add_store_node、add_compute_node、add_constant_node、add_cast_node — 均在 EpilogueVisitorGraph 中定义（evg_extension.py:147-234）。

                        • 约束：所有节点名称唯一（check_not_exist 断言）；return 不能直接返回表达式（SyntaxError）；未定义变量使用时报错。

                        • 输出：完整 networkx DiGraph，节点类型为 NodeBase 子类（LoadNode/ComputeNode/StoreNode/CastNode/ConstantNode），边带 weight 表示操作数位置。

                        阶段三：图优化 Pass 管线——7 道 Pass 顺序执行

                        执行流程分析

                        • 输入：原始 DAG，包含各类 IR 节点（LoadNode, ComputeNode, StoreNode, CastNode, ConstantNode）。

                        • Pass1 去重+死节点消除（evg_definition.py:21）：① 先消除 ConstantNode→StoreNode 模式（无意义的常量写回）；② 构建节点签名（类型+算子+有序输入名），遍历拓扑序，检测签名完全相同的节点并合并——保留第一个，把后续节点的出边重连到保留节点，删除重复节点；③ 从 output StoreNode 出发反向 DFS 收集可达节点，删除不可达的死节点。使用 Counter 比较输入集合是否一致（evg_definition.py:100）。

                        • Pass2 标量运算折叠（evg_definition.py:152）：遍历 ComputeNode，检查输入中是否有 ConstantNode。若 fn 在 EpilogueOpVectorToScalar 映射中（Add→Adds, Mul→Muls），则将算子替换为标量版本，记录标量值，删除 ConstantNode。

                        • Pass3 形状推导（evg_definition.py:168）：按拓扑序遍历，收集每个节点的有序输入 shape，应用 Numpy 广播规则——右对齐，从右到左逐维比较，1 可与任意维匹配，否则必须相等。维度冲突抛 RuntimeError。

                        • Pass4 广播传播（evg_definition.py:231）：检测 src_shape 到 dst_shape 是否需广播。右对齐逐维检查：若 dst 最右维需广播标记为列广播并抛异常（当前不支持 ColumnBroadcast，evg_definition.py:311）；若其他维需广播标记为 RowBroadcast，shape 为 1D 时 reshape 为 (1, N)。

                        • Pass5 动态形状（evg_definition.py:320）：检查 shape 中是否有 sympy.Symbol 类型实例，若有则使用预处理好的 symbol_shape_substitution_dict 替换为 m/n 字符串。

                        • Pass6 Impl 匹配（evg_definition.py:337）：每个节点调用 get_impl() 绑定 C++ 实现类（node.py:65）。LoadNode 可能为 AccLoadImpl/AuxLoadImpl/RowBroadcastImpl；StoreNode 可能为 AuxStoreImpl/NoOpImpl；ComputeNode 可能为 ComputeImpl/ScalarComputeImpl。绑定后消除 NoOpImpl 节点——将输入直连到输出，删除中间节点。

                        • Pass7 DAG→Tree 融合（evg_definition.py:360）：检测 out_degree > 1 的多输出节点，对每个多输出节点执行 _find_lca——从各输出分支做前向 DFS，收集可达节点集合，取交集的最小拓扑序节点为 LCA（Least Common Ancestor）。融合 LCA 和分支节点为 TopoVisitorNode，子图存为独立的 EpilogueVisitorGraph。递归执行直到无多输出节点。

                        • 输出：优化后 DAG，所有 IR 节点已绑定具体 C++ 实现类，拓扑关系明确，可直接用于代码生成。

                        阶段四：C++ 模板代码生成

                        执行流程分析

                        • 类型定义生成（EVGDef.definition()，evg_definition.py:476）：拓扑序遍历所有节点：

                          • def_node()：每个非 disabled 节点通过 node.impl.type_decl 生成类型声明。例如 AccLoadImpl 生成 using AccLoad = Catlass::Epilogue::Fusion::VisitorAccLoad<FLOAT>;（node_impl.py:89）。

                          • def_tree_node()：对有输入（in_degree > 0）的节点生成 using EVGNodeName = TreeVisitor<NodeType, EVGChild1, ...>（evg_definition.py:503）。子节点按 sorted_inputs 顺序排列。

                          • def_subgraph_node()：对 TopoVisitorNode 生成 using NodeName = TopologicalVisitor<EdgeTuple, Node1, Node2, ...>（evg_definition.py:520）。EdgeTuple 是 tla::tuple<tla::seq<child_idx...>, ...>，子图中每个节点的依赖按拓扑序号索引。

                          • callback_name 取最后一个拓扑节点的 visitor 名（evg_definition.py:491）。

                        • 参数初始化生成（EVGArg，evg_definition.py:556）：从 output StoreNode 出发反向递归生成嵌套大括号初始化表达式。TreeVisitor 参数为 {{child_args...}, node_args} 嵌套结构。TopoVisitor 参数为 {node0_args, node1_args, ...} 按子图拓扑顺序。

                        • compute_length 计算（evg_definition.py:592）：遍历所有 storage 节点（非 StoreNode、非 disabled），按元素类型分组计数。公式：computeLength = 216 * 1024 / Σ(num_i * sizeof(element_i)) / 2 / 32 * 32。其中 216KB = ArchTag::UB_SIZE（Ascend 架构 UB 总大小），/2 为双缓冲预留一半空间，/32*32 确保 32 字节对齐。最终结果作为 Int<computeLength> 模板参数，控制向量分块粒度。

                        • 输出：(callback_name, evg_args, evg_str, arg_renames) 四元组。evg_str 包含所有 using 类型别名定义，evg_args 包含 typename EVGXxx::Arguments evg_args{...};。最终嵌入到 BasicMatmulTlaVisitorKernel._EVG_TEMPLATE（basic_matmul_tla_visitor.py:97）形成完整 kernel 源码。

                        阶段五：Kernel 编译与跨核调度——AIC 矩阵乘 + AIV 后处理

                        执行流程分析

                        • 输入：Params 包含 problemShape (m, n, k)，A/B/C 矩阵 GM 地址及 layout，epilogueParams。Workspace 布局：[C矩阵: m*n*sizeof(ElementC) 字节] + [EVG 各算子 workspace: per-op 32 字节对齐]。

                        • BlockScheduler（basic_matmul_tla_visitor.hpp:123）：根据 L1_TILE_M/N 计算 tile 循环次数。GetBlockIdx() 获取当前核索引，AIV 核除以子核数 GetSubBlockNum() 得到 8 核中的逻辑索引。loopStep = GetBlockNum() 实现多核间 tile 分摊。

                        • L2 Cache 优化（basic_matmul_tla_visitor.hpp:139-144）：当 CeilDiv(m, L1_TILE_M) == 1 时 A 矩阵只有一行 tile，B 矩阵 Disable L2 Cache 避免无效缓存。同理对 A。

                        • Workspace 模式：GM workspace——BasicMatmulTlaVisitor 的 USE_UB_WORKSPACE=false（basic_matmul_tla_visitor.hpp:46）。AIC 将 MMAD 结果通过 TLA tile copy 写入 GM（tensorBlockC 对应 GM workspace）。UB workspace——BasicMatmulTlaUbVisitor 的 USE_UB_WORKSPACE=true（basic_matmul_tla_ub_visitor.hpp:47），AIC 将 MMAD 结果直接写入 UB 的 [0, L0C_SIZE/2) 区域，需额外跨核同步（0x4 模式 flag）。

                        • 跨核同步：GM workspace 用 CrossCoreSetFlagWithReverse<0x2>（basic_matmul_tla_visitor.hpp:194）和 CrossCoreWaitFlagWithReverse<0x2>（basic_matmul_tla_visitor.hpp:196）。UB workspace 用 0x4 模式（basic_matmul_tla_ub_visitor.hpp:194-205）：AIC 需要等待 AIV 完成（CrossCoreWaitFlag<0x4>），MMAD 后设置 flagAicFinish、flagAicFinishNotifyAiv1（AIV1 用偏移 +16 的 flag 同步）。

                        • 输出：每个 loop 步骤中 AIC 完成 MMAD → AIV 执行 epilogue。全部循环后 PipeBarrier<PIPE_ALL>() 全局同步。

                        阶段六：BlockEpilogue 双缓冲流水线——LOAD-COMPUTE-STORE 三级流水

                        执行流程分析

                        • 输入：tensorBlockC（GM workspace 中一个 tile 的 C 矩阵视图）、blockShape/blockCoord/actualBlockShape（来自 BlockScheduler）。

                        • Subblock 划分（block_epilogue_visitor.hpp:122-129）：subblockShape.row = CeilDiv(actualBlockShape.row(), GetSubBlockNum())，将 tile 行方向均分给 AIV 各子核。subblockCoord = {GetSubBlockIdx(), 0}，actualSubblockShape = Min(subblockShape, remaining) 处理边界 tile。

                        • UB 起始偏移（block_epilogue_visitor.hpp:144）：GM workspace（USE_UB_WORKSPACE=false）：ub_offset0 = 0，全部 UB 空间（216KB）可用于 EVG 缓冲。UB workspace（USE_UB_WORKSPACE=true）：ub_offset0 = L0C_SIZE / 2，前 [0, L0C_SIZE/2) 给 MMAD 的 C tensor，后一半给 EVG。

                        • 双缓冲初始化（block_epilogue_visitor.hpp:61-78）：构造函数中分配两套硬件事件（eventVMTE2[2], eventMTE2V[2], eventMTE3V[2], eventVMTE3[2]），初始设置 V_MTE2[0/1] 和 MTE3_V[0/1] 事件 Flag（表示初始状态就绪）。两套 callbacks 各自有独立的 UB 偏移。

                        • Tile 循环策略（block_epilogue_visitor.hpp:159）：

                          • 若 cols <= COMPUTE_LENGTH（列宽 ≤ 向量计算长度）：一次处理多行。maxRowsPerTile = COMPUTE_LENGTH / RoundUp(cols, BYTE_PER_C0)，tileRows = min(remainRows, maxRowsPerTile)。防零保护。

                          • 若 cols > COMPUTE_LENGTH（列宽 > 向量计算长度）：一次一行，每行切分列。tileCols = min(remainCols, COMPUTE_LENGTH)，逐列块推进。

                          • ubListId = 1 - ubListId 实现双缓冲轮转。

                        • 对齐：RoundUp<BYTE_PER_C0>(cols) 确保列对齐到 C0 单元（BYTE_PER_C0 = 32 字节）。

                        阶段七：单 Tile 三级流水——LOAD→COMPUTE→STORE 硬件同步

                        执行流程分析

                        • LOAD 阶段（block_epilogue_visitor.hpp:233-235）：WaitFlag V_MTE2[ubListId] 等待前一轮 COMPUTE 完成（第一轮已由构造函数预设）。执行所有算子的 visit<LOAD, ...>：

                          • VisitorAccLoad：GM workspace 时用 CopyGm2UbTla 从 GM 拷贝 C 矩阵到 UB（visitor_acc_load.hpp:100）。UB workspace 时跳过加载，直接计算 UB 偏移指向已有数据（visitor_acc_load.hpp:106）。

                          • VisitorAuxLoad：用 CopyGm2UbTla 从 GM 加载辅助输入（bias/scale 等），使用 globalOffset 定位 GM 中的 tile 位置（visitor_aux_load.hpp:100-112）。

                          • VisitorRowBroadcast：LOAD 阶段仅加载第一行 1xN 到 UB（visitor_row_broadcast.hpp:94-114）。

                          • 完成后 SetFlag MTE2_V[ubListId] 通知 COMPUTE。

                        • COMPUTE 阶段（block_epilogue_visitor.hpp:237-241）：WaitFlag MTE2_V（等待加载完成）+ WaitFlag MTE3_V（等待前一轮 STORE 完成）。执行 visit<COMPUTE, ...>：

                          • VisitorCompute：构造 ComputeFn<ElementCompute> 聚合对象（带标量参数），调用 compute_fn(ubOut, compute_length, inputs...)（visitor_compute.hpp:116）。完成后 PipeBarrier<PIPE_V>() 保证向量计算管道排空。

                          • VisitorCast：调用 AscendC::Cast 进行数据类型转换（visitor_cast.hpp:70），支持 CAST_NONE/CAST_ROUND/CAST_ROUND_AWAY_FROM_ZERO 等舍入模式。

                          • VisitorRowBroadcast：COMPUTE 阶段将已加载的第一行用 DataCopy 逐行复制到 MxN（visitor_row_broadcast.hpp:117-123）：ubOut[r * alignedCols] → ubOut[0]，长度为 alignedCols 元素。

                          • 完成后 SetFlag V_MTE2（通知 LOAD）+ SetFlag V_MTE3（通知 STORE）。

                        • STORE 阶段（block_epilogue_visitor.hpp:243-245）：WaitFlag V_MTE3 等待 COMPUTE 完成。执行 visit<STORE, ...>：

                          • VisitorAuxStore：用 CopyUb2GmTla 将 UB 计算结果写回 GM（visitor_aux_store.hpp:89-114）。使用 params_ptr->ptr_aux 输出地址 + params_ptr->layout GM layout，配合 globalOffset 拼装目标 tile 坐标。若 ptr_aux == nullptr 跳过（支持选择性写回）。

                          • 完成后 SetFlag MTE3_V[ubListId] 通知下一轮 COMPUTE 该缓冲可用。

                        • 双缓冲硬件事件流水：四个硬件事件（V_MTE2/MTE2_V/MTE3_V/V_MTE3）形成闭环依赖链。两组事件（[0] 和 [1]）由 ubListId 交替使用，实现 LOAD-COMPUTE-STORE 三级重叠流水。Destructor 等待所有未完成的事件（block_epilogue_visitor.hpp:84-89）。

                        阶段八：Visitor 调度器——TreeVisitor 与 TopologicalVisitor

                        执行流程分析

                        • TreeVisitor 调度（tree_visitor.hpp:19-105）：using EVG = TreeVisitor<NodeOp, ChildOps...>（tree_visitor.hpp:108）。visit() 接收变长参数 args（来自上游的输入）。首先 collect_child_outputs 用 tla::make_seq<Rm1>（Rm1 = 子节点数）展开索引序列，依次调用每个子节点的 tla::get<Is>(callbacks_tuple).template visit<Stage, ArchTag>(tensorTile, ...)，子节点输出收集为 tla::tuple<Child0Output, Child1Output, ...>。然后 call_parent_with_outputs 将子节点输出元组展开，传给最后一个 Op（父节点）的 visit 回调。父节点输出类型 ElementOutput 定义为 NodeOp::ElementOutput（tree_visitor.hpp:24）。

                        • TopologicalVisitor 调度（topological_visitor.hpp:37-175）：EdgeTuple 形如 tla::tuple<tla::seq<child0_idx...>, tla::seq<child1_idx...>, ...>，长度等于 Ops 数量，最后一个 Op 为根节点。visit() 从根节点 R-1 开始，沿 EdgeTuple 指定的子索引递归 DFS。visited_[] 数组标记节点是否已访问，cache_[] 缓存输出 LocalTensor，重复访问时直接返回缓存值（topological_visitor.hpp:76-77）。reset_flags() 每个 tile 开始时重置所有 visited 标记。

                        • UB 偏移管理：每个算子 get_callbacks 从 resource.ubBuf 分配 UB 空间并前进 ub_offset += compute_length * sizeof(Element)（visitor_acc_load.hpp:121，visitor_compute.hpp:129）。compute_length 由 Python 生成的 constexpr uint32_t computeLength 决定。assert(ub_offset <= ArchTag::UB_SIZE) 确保不超界。

                        • 算子实现（operations.hpp）：

                          • 一元：Exp/Relu/Sqrt/RsqrtFast/LeakyRelu/Silu/Sigmoid/Reciprocal → 调用 AscendC::Exp/Relu/Sqrt/... 硬件指令

                          • 二元：Add/Mul/Sub/Div/Max/Min → AscendC::Add/Mul/Sub/...，支持变参链式 if constexpr (sizeof...(rest) > 0) 递归

                          • 标量：Adds/Muls → AscendC::Adds/Muls(dst, src, scalarValue, compute_length)

                          • 融合：AddRelu → Add + PipeBarrier + Relu；Sigmoid → Muls(-1) + Exp + Adds(1) + Divs(1/)

                          • Cast：AscendC::Cast(dst, src, RoundMode, compute_length)

                          • 条件编译：Sigmoid/Prelu/Reciprocal/Rsqrt 仅在 CATLASS_ARCH == 3510（Ascend950）下可用（operations.hpp:268-325）

                        阶段九：UB Workspace 专有路径——Ascend950 直接 UB 读写

                        执行流程分析

                        • UB Workspace 核心区别（basic_matmul_tla_ub_visitor.hpp）：MMAD 的 C 矩阵结果直接写入 UB 而非 GM。AIC 和 AIV 共享 UB 空间，因此需要更严格的同步机制。

                        • GetMmadUbTensor（block_epilogue_visitor.hpp:93-106）：返回 UB 中以偏移 0 起始的 ElementC* 缓冲，layout 使用 strideC = RoundUp(n, BYTE_PER_C0)，创建 LocalTensor 供 MMAD 写入。该函数仅 USE_UB_WORKSPACE=true 时可用。

                        • 跨核同步（basic_matmul_tla_ub_visitor.hpp:153-206）：

                          • AIV 先发信号（basic_matmul_tla_ub_visitor.hpp:153）：进入循环前 AIV 设置 flagAivFinish，表示 AIV 初始状态就绪，AIC 可以开始第一轮 MMAD。

                          • AIC 等待 AIV（basic_matmul_tla_ub_visitor.hpp:181-182）：每次 MMAD 前等待 flagAivFinish 和 flagAiv1FinishNotifyAic（AIV1 用偏移 +16），确保前一轮 epilogue 完成。

                          • AIC→AIV 通知（basic_matmul_tla_ub_visitor.hpp:194-195）：MMAD 完成后设置 flagAicFinish（通知 AIV0）+ flagAicFinishNotifyAiv1（通知 AIV1，偏移 +16）。

                          • AIV 等待 AIC（basic_matmul_tla_ub_visitor.hpp:199）：CrossCoreWaitFlag<0x4, PIPE_V>(flagAicFinish) 等待 MMAD 完成，然后执行 epilogue。

                          • AIV→AIC 通知（basic_matmul_tla_ub_visitor.hpp:205）：epilogue 完成后 CrossCoreSetFlag<0x4, PIPE_V>(flagAivFinish) 通知 AIC 可以下一轮。
